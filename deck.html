Correlation Rules

Wazuh correlate events từ nhiều nguồn — cùng attack nhưng qua Snort và ModSecurity sẽ được gom thành 1 unified alert thay vì 2 alert riêng biệt

Wazuh Rule Example

<!-- Detect brute force: 5 failed auth in 60s -->
<rule id="100200" level="10">
  <if_matched_sid>100100</if_matched_sid>
  <same_srcip />
  <timeframe>60</timeframe>
  <frequency>5</frequency>
  <description>Brute force attack detected</description>
</rule>

Testing Methodology

3 giai đoạn kiểm thử

1

Automated Vulnerability Scanning  Acunetix

Scan tự động để lập baseline lỗ hổng trước khi deploy monitoring stack. Xác định attack surface của web application.

2

Simulated Attacks  BurpSuite + Manual

Thực hiện SQLi, XSS, brute force, directory traversal có kiểm soát trong môi trường lab. Verify end-to-end từ tấn công đến Telegram notification.

3

Log Analysis & Rule Tuning  Wazuh Dashboard

Phân tích Wazuh alerts và Snort/ModSecurity logs từ Phase 2. Xác định false positive, tune rules, validate detection accuracy.

Lessons Learned

Nếu làm lại — sẽ khác gì?

Điểm yếu phát hiện được

  • Snort FP rate cao với HTTPS — cần whitelist tuning
  • Lab VMware không capture production workload đặc trưng
  • Wazuh rules default cần customize nhiều hơn
  • Không test slow-and-low attacks (evasion techniques)

Nếu scale lên production

  • Kubernetes-native agents thay vì bare-metal VMs
  • ELK Stack thay Zabbix cho log analytics
  • SOAR integration để tự động response
  • Threat intelligence feed cho Snort/Wazuh rules
  • Multi-tenant: tách môi trường dev / staging / prod
